# 2026-07-10 本地优先产品全面收口设计

> 历史快照：本文对应 `1.2.0+2` 的收口设计，其中 4 Tab 是当时决策。当前基线为 `1.4.0+4` 加 `Unreleased` 改动，移动端已经采用今日、AI、发现、监控、我的 5 Tab；当前事实请查看 [产品、数据与系统边界](../../plans/product_ia_data_plan.md) 和 [README](../../../README.md)。

## 背景

当前项目已经具备真实 AI/GitHub 数据读取、SQLite 缓存、本地历史快照、
Riverpod 状态管理、Windows Release 构建和 204 个自动化测试，但产品文案、
监控语义、数据来源表达、版本文档和部分架构边界没有完全对齐。

本设计的目标不是扩张产品范围，而是让以下三件事一致：

1. 产品对用户承诺的能力。
2. 客户端在无服务端条件下实际执行的能力。
3. UI 展示的数据来源与计算口径。

## 已确认约束

- 继续以 Windows 桌面端为当前实现优先级。
- 不新增后端、云同步、服务端定时任务或邮件推送服务。
- GitHub 和 AI 资讯继续由客户端直连公开 API。
- 应用必须在网络失败时保持可用。
- 用户内容继续保存在本机；GitHub Token 继续使用 Secure Storage。
- 手机端只收敛为既定 4 Tab，不扩张新的移动端业务页面。
- 所有功能变更采用测试驱动，最终必须通过格式化、静态检查、全量测试和
  Windows Release 构建。

## 方案比较与决策

### 方案 A：本地优先客户端收口（采用）

把“监控”定义为应用启动或用户刷新时执行的前台检查；实现真实规则计算、
告警持久化和来源标识，同时删除邮件、跨设备和后台推送等不成立的承诺。

优点：与现有架构一致，可离线、可测试、无需新增运维系统。

代价：应用关闭后不会继续产生新告警。

### 方案 B：常驻桌面进程

新增托盘常驻、定时抓取和 Windows 本地通知。

优点：在电脑开机且进程运行时可以持续检查。

代价：涉及生命周期、安装器、自启动、功耗、系统权限和异常恢复；仍不能提供
真正的跨设备或离线电脑上的持续监控。

### 方案 C：服务端监控平台

新增账号系统、调度器、持久任务、通知投递和云同步。

优点：可以实现真正的持续监控和跨设备能力。

代价：直接违反当前“无后端阶段”约束，属于新的产品阶段。

因此本轮采用方案 A。方案 B/C 不作为隐藏待办混入客户端代码。

## 总体交付结构

工作拆为四个可独立验证的批次：

1. 能力口径、版本和导航收口。
2. 前台监控规则、观测历史和告警持久化。
3. 数据来源与缓存可信度统一。
4. 架构边界、文件体量、测试和文档最终收口。

每批完成后单独运行目标测试和 `flutter analyze`；全部批次完成后再运行全量
验证与 Windows Release 构建。

## 批次一：能力口径、版本和导航

### 产品文案

统一使用以下能力定义：

- “仓库监控”改为“仓库跟踪”或在副标题明确“打开应用时检查”。
- “实时”仅用于本次远端响应，不表示后台持续运行。
- 删除“邮件摘要”“每日报告”“周报推送”“跨设备访问”“登录后同步”等
  当前无法兑现的承诺。
- GitHub 授权只承诺提升 API 配额、读取公开资料和免手工粘贴 Token。
- OAuth Client ID 未配置时，登录入口显示为不可用配置项，不诱导用户进入必然
  失败的流程。
- 通知设置页只保留应用内告警中心相关选项；不支持的渠道不以可操作开关出现。

### 导航

- 桌面端保留 8 个入口：总览、AI 动态、GitHub 热榜、AI 雷达、发现、仓库
  跟踪、深度报告、设置。
- compact 布局只显示 4 个底部入口：今日、AI、项目、设置。
- 4 个移动入口映射到现有分支：`home`、`ai_news`、`trending`、`profile`。
- AI 雷达、发现、跟踪和报告在 compact 下作为“项目”域内页面访问，不再各占
  一个底部位置。

### 版本和文档

- 新功能版本统一升级为 `1.2.0+2`。
- README、RUN、产品数据方案、CHANGELOG 与实际 8 入口、分级 TTL、无后台
  监控边界保持一致。
- 删除已失效的 1GB 容量守卫、侧栏持久化和完全服务端化暗示。

## 批次二：前台监控闭环

### 领域模型

新增三个独立概念：

1. `MonitorObservation`：某仓库在某次成功远端检查时的 stars、forks、
   openIssues 和 observedAt。
2. `MonitorRule`：稳定规则 ID、是否启用、阈值和计算方式。
3. `MonitorAlertEvent`：稳定 ID、仓库、规则 ID、指标、当前值、阈值、严重度、
   observedAt、readAt 和 archivedAt。

告警事件不再由页面临时拼装；页面只读取持久化事件。

### 观测存储

- 新增 `MonitorObservationDao`，基于通用 JSON Snapshot DAO 保存每个仓库的
  最近 90 个按日观测点。
- 同一仓库同一天重复刷新时覆盖当天观测，不制造伪造的日内增长。
- 远端请求成功后才写入观测；缓存命中、种子回退和请求失败不写入观测。
- 初次观测只建立基线，不产生增长类告警。

### 规则语义

沿用现有四条规则，但把计算方法固定下来：

- `star_daily_delta`：当前 Star 减去上一不同日期观测，达到 200。
- `star_daily_rate`：Star 日增量除以上一观测 Star，达到 10%。
- `fork_daily_delta`：当前 Fork 减去上一不同日期观测，达到 50。
- `issue_heat_ratio`：`(当前 openIssues + 1) / (上一观测 openIssues + 1)`
  达到 5 倍。

所有增长值按两个不同本地日期的观测计算，不把估算曲线用于告警。

### 告警生成和去重

- 每次远端监控刷新完成后，由纯 Dart `MonitorRuleEvaluator` 计算规则。
- 稳定事件 ID 使用 `repo + ruleId + observedLocalDate`，同一天重复刷新不重复告警。
- 单个仓库请求失败不会阻止其他仓库完成检查。
- 告警最多保留 500 条，按时间淘汰最旧记录。
- 现有已读、归档和恢复操作迁移到告警事件存储；无法匹配的旧状态安全丢弃。

告警事件使用独立 SQLite 表 `monitor_alert_event` 持久化，字段固定为
`id/repo_full_name/rule_id/metric/value/threshold/severity/observed_at/read_at/archived_at`。
数据库 schema 升级到 v4。该表属于用户内容，不加入“清理缓存”的删除表清单。

### 刷新行为

- 页面首次进入：新鲜缓存优先，不产生新观测或告警。
- 用户手动刷新：删除当前监控缓存，访问远端，成功后执行规则。
- TTL 到期自动读取：访问远端，成功后执行规则。
- GitHub 限流：使用陈旧缓存或种子，不执行规则，并展示来源状态。

## 批次三：数据来源与可信度

### 两个维度

当前 `DataProvenance` 混合了“数据从哪里来”和“指标怎样计算”。本轮明确拆成：

- `DataFreshness`：live、freshCache、staleCache、seed。
- `MetricBasis`：observed、estimated、seed。

远端/缓存描述整份响应的新鲜度；观测/估算描述单个趋势指标的计算方式。这样可以
准确表达“新鲜缓存中的估算曲线”或“陈旧缓存中的真实历史”。

### 应用方式

- Repository 返回 `DataResult<T>`，包含 data 和 freshness。
- Repo、语言和热点模型只保留 metric basis，不再把缓存状态写进业务数值实体。
- 页面头部统一展示 freshness；趋势图和增长数字旁展示 metric basis。
- AI 动态、热榜、AI 雷达、发现、监控、仓库详情和深度报告全部接入。
- 缓存损坏时删除当前 key 并继续远端请求；远端仍失败时进入种子或错误状态。

### ETag

- 只对单 URL 资源启用 ETag，不给多个请求聚合出的 digest 伪造整体 ETag。
- 首批接入 Repository detail、Contributors 和公开 User 请求。
- 单资源 payload 使用 `resource:<tokenScope>:<method>:<url>` 作为
  `json_snapshot_cache` key，ETag 继续存放在对应 `cache_meta.payload_hash`。
- 304 时复用对应资源缓存并刷新获取时间，不重新解析空响应。
- Search 聚合继续使用 TTL，直到具备逐 URL 缓存模型。

## 批次四：架构和维护性

### 跨 Feature 依赖

- 新增 Core 级 `RepositoryFeed` 契约，向报告模块提供仓库候选、趋势和语言分布。
- Trending 在 DI 组合层适配为 `RepositoryFeed`。
- Project 只依赖 Core 契约，不再直接 import Trending application/domain。
- Home 作为应用级聚合页面保留读取多个 Feature Provider 的职责，并在架构文档中
  明确这是组合根例外。

### 文件拆分

除 i18n 映射和纯种子数据外，生产 Dart 文件控制在 300 行内。优先拆分：

- Discover Repository：查询、Profile、Codec 分离。
- Login Page：状态内容和结果组件分离。
- Profile Settings Card：主题、启动、数据源区块分离。
- Router：一级分支和辅助路由分离。

拆分只改变组织结构，不改变公开行为。

### 缓存和用户数据边界

- “清理缓存”只删除可重新生成的网络缓存和趋势观测，不删除收藏、关注、规则、
  告警事件或 Token。
- “重置本地数据”如未来需要，必须作为独立危险操作实现；本轮不增加。
- 继续保留无限容量显示，但对历史和告警分别设置数量上限，避免无界增长。

## 错误处理

- 网络错误：新鲜/陈旧缓存优先，随后种子或差异化错误页。
- GitHub 限流：触发全局 RateLimitGate，其它 Repository 在窗口内直接走 fallback。
- 单项聚合失败：记录日志并保留成功项；全部失败才整体回退。
- 缓存解析失败：删除坏 key，不把解析异常伪装成网络异常。
- 告警存储失败：不影响仓库数据展示，但页面显示本地保存失败提示，并允许重试。
- OAuth 未配置：入口禁用并说明原因，不发起无效请求。

## 测试设计

### 单元测试

- 四条监控规则的未达到、刚好达到、超过阈值。
- 首次观测不告警、同日覆盖、跨日计算、事件去重和 500 条上限。
- DataFreshness 与 MetricBasis 的序列化和 UI 标签。
- ETag 200/304/缓存损坏/无缓存 304 的行为。
- RepositoryFeed 适配和 Project 解耦。

### Provider/Repository 测试

- 手动刷新只失效当前监控 key。
- 缓存命中不生成观测。
- 远端成功才执行规则。
- 限流和远端失败不产生伪告警。
- 告警已读、归档和恢复可跨 Provider 重建保存。

### Widget 与布局测试

- 桌面 8 入口和 compact 4 入口。
- 不支持的邮件/跨设备操作不再显示为可用入口。
- 1366x768、1024x768、390x844 下无 overflow。
- 来源与指标口径 badge 在关键页面可见。

## 验收标准

全部条件满足才算完成：

1. UI 和文档不再声称后台推送、邮件或跨设备同步。
2. 桌面 8 入口、compact 4 入口与文档一致。
3. 用户规则基于真实跨日观测生成持久化告警，刷新不会重复。
4. 所有远端主链路能区分实时、新鲜缓存、陈旧缓存和种子。
5. 趋势数字能区分观测与估算。
6. Project 不再直接依赖 Trending Feature。
7. 版本统一为 1.2.0+2，README、RUN、产品方案和 CHANGELOG 同步。
8. 全量格式化、静态检查、测试和 Windows Release 构建通过。
9. 工作区最终只包含本任务的有意变更，并提交到活动分支。

## 非目标

- 不新增服务端、登录账号数据库或云同步。
- 不实现邮件、短信或第三方推送。
- 不实现 Windows 托盘常驻、自启动或后台调度。
- 不引入 GH Archive、Reddit、X 或 Hacker News 新数据源。
- 不重做整体视觉设计。
- 不创建新的移动端业务页面。
