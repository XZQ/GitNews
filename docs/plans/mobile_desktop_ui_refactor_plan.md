# 移动端与桌面端 UI 整改基线

更新时间：2026-07-18
状态：已完成（2026-07-18 双端验收）
依据：当前代码、2026-07-18 Android 真机完整滚动审查、现有产品与数据边界

## 1. 整改原则

1. 不维护两套业务代码：数据、领域模型、Riverpod 状态、路由语义、主题 Token 和基础组件共用。
2. 不强行共用整棵页面：总览、资讯详情、仓库详情、监控详情和“我的/Profile”分别编排移动端与桌面端。
3. 移动端优先完成日常阅读、发现、监控和个人内容任务；桌面端保持高密度、可扫描的情报工作台。
4. 不把种子、估算或缺失数据伪装成实时结果；真实零值与未知值分开表达。
5. 不以更新 Golden 作为视觉验收；关键页面必须在当前真机和 Windows 可见窗口中检查。

## 2. 导航边界

### 移动端与紧凑窗口

固定五个一级目的地：总览、AI、发现、监控、我的。热榜、AI 雷达、详情、提醒、收藏、关注、规则和高级设置保留为二级页面。

### 桌面端

保留八个一级目的地：总览、AI 动态、GitHub 热榜、AI 雷达、发现、仓库监控、深度报告、设置。详情继续在应用壳的内容区域内打开。

## 3. 页面编排矩阵

| 页面 | 移动端 | 桌面端 | 共享边界 |
|---|---|---|---|
| 总览 | 单列摘要、重点优先、长区块进入二级页 | 多区块高密度工作台 | 数据与内容区块组件 |
| AI 列表 | 紧凑页头、单列时间流 | 搜索筛选常驻、宽列表 | 查询、筛选、分页、卡片语义 |
| AI 详情 | 单页上下阅读、底部动作 | 内容主栏与辅助信息区 | 语言判断、内容状态、收藏反馈 |
| 发现 | 单列分类与列表 | 多列发现与筛选 | 数据、分类、实体卡信息 |
| 监控主页 | 状态摘要、仓库列表、告警入口 | 仓库表、规则与告警双栏 | 监控状态、规则、告警 |
| 监控详情 | 仓库、趋势、告警单列 | 趋势与告警并排 | 趋势卡、告警卡和数据口径 |
| 仓库详情 | 2×2 指标、纵向详情 | 4 列指标、主辅双栏 | 仓库实体、图表和活动组件 |
| 我的/Profile | 用户中心，普通设置优先 | 设置导航与详情工作区 | 收藏、关注、偏好和安全存储 |
| 简单二级列表 | 全屏返回、单列 | 内容区列表或主从布局 | 数据、行组件和页面状态 |

## 4. 数据可信度约定

- `live`、`freshCache`、`staleCache`、`seed` 表达响应新鲜度。
- `observed`、`estimated`、`seed` 表达指标口径。
- 缓存中的真实指标仍为 `observed`，不能因为缓存而变为 `estimated`。
- 没有趋势数据时显示空状态，不在展示层生成看似真实的曲线。
- `0` 只用于服务端或本地事实明确为零；未知值使用暂无数据或隐藏。
- 所有关键数据说明来源、更新时间或降级原因。

## 7. 第一条验收切片

- Android 真机从页面顶部滑到末尾，无黄色溢出条、裁切或底栏遮挡。

- 600–1024 px 使用中等布局，不出现桌面双栏挤压。
- Windows Expanded 布局保持 4 列指标和主辅双栏。
- 长仓库名、长描述、大数字、空趋势和空告警均有稳定状态。
- 浅色和深色主题可读。
- `dart format`、`flutter analyze`、针对性测试和全量测试通过。
- 桌面影响改动通过 Windows Release 构建和可见窗口烟测。

## 8. 完成定义

单个整改项只有在代码、针对性回归测试和对应平台的当前运行证据同时成立时才标记完成。移动端通过不代表桌面端通过，更新 Golden 也不代表设计验收通过。
